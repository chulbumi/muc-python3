# `무공` 명령 이관 현황

기준 파일은 `cmds/무공.py` 하나이며, Rust에 있던 무공 사용/성공 문구와
다른 플레이어 조회 “구현 중” 문구는 Python에 없으므로 제거했다.

## 현재 재검증 범위

- 인자가 없으면 본인의 무공을 출력한다.
- 관리자등급 1000 이상이고 인자가 있을 때만 같은 방 객체를 조회한다.
- 일반 사용자의 인자는 Python처럼 무시하고 본인의 무공을 출력한다.
- 아이템 또는 찾지 못한 대상을 지정하면 Python의 안광 문구를 출력한다.
- `초급`부터 `기타`까지 Python `Body.skillLvType` 순서와
  `data/config/murim.json`의 `메인설정/*무공` 순서를 사용한다.
- `skillMap` 숙련도를 우선하고, `skillList`에만 있는 무공은 1성으로 표시한다.
- 각 셀은 Python `fillSpace(20, ...)`와 같은 EUC-KR 바이트 폭이며 3열로 배치한다.
- `비전수련`, `비전이름`, ANSI, 경계선, CRLF를 Python 출력과 대조했다.
- `무공이름`, `무공숙련도`, `비전이름`은 저장 시 Python JSON 배열로 기록한다.
- Rust built-in `무공`과 Python에 없는 `기술보기` 등록을 제거하여
  핫리로드되는 `cmds/무공.rhai`가 실제 명령 핸들러가 되도록 했다.

## 남은 구조적 제약

Python `Room.objs`는 player/mob/item을 하나의 삽입 순서로 보존하지만 현재 Rust는
각 종류를 별도 저장한다. 이름 또는 반응이름이 같은 서로 다른 객체가 한 방에
동시에 존재하면 Python의 “최초 객체”를 증명할 수 없다. 이 경우 임의 종류
우선순위를 만들지 않고 대상 미발견으로 처리한다. 여러 몹 중 숫자로 N번째를
고르는 경우도 같은 이유로 임의 순서를 사용하지 않는다. 통합 방 객체 순서가
이관되면 `get_mugong_view`의 충돌 처리도 그 순서를 사용하도록 바꿔야 한다.
