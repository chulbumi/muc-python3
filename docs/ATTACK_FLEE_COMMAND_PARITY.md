# `쳐` / `도망` Python 동작 대조 범위

기준 소스는 `cmds/쳐.py`, `cmds/도망.py`, `objs/room.py`의
`findObjName`/`getRandomExit`, `objs/player.py`의 `setFight`/`enterRoom`이다.

현재 재현한 범위:

- `쳐`의 검사 순서와 원문 문구: 빈 대상, `전투금지`, `시체`, 객체 종류,
  `몹종류`, 사망/리젠 상태, 이미 공격 중, 다른 대상과 전투 중
- 같은 방 몹의 이름·반응이름·숫자 순번 조회와 `setFight`의 기본 상호 대상,
  전투 상태, 기본 전투 시작 문구, 무공이 없는 `전투종류` 1/2 덩달이 몹
- `도망`의 전투/방 속성/1초 `_runaway` 검사, 능파미보 상태 보너스 자료,
  대상 레벨·민첩 공식과 `randint(0, 100)` 양 끝 포함
- Python `sortExit` 순서, 숨겨진 출구 선택 실패, 다중 목적지의 별도 난수 선택,
  레벨·인원·성격·방파 제한과 안전하게 표현 가능한 `enterRoom(..., "도망")`
- 출발/도착 알림은 전 접속자 목록이 아니라 `get_room_players(ob)`만 사용

현재 Rust 모델이 Python 상태를 충분히 보존하지 못해 이동/전투 성공으로
처리하지 않는 분기:

- `setFight` 안에서 즉시 실행되는 자동무공 및 몹 공격/방어 무공
- 시작된 전투의 매초 `Player.doFight()` 연속 처리와 타 플레이어 prompt 전송
- 목적지 `Room.update()` 중 만료 바닥 아이템, 복잡한 몹 상태/대화/아이템 리젠
- 상호작용·재이동을 포함하는 입장 이벤트, 선공몹 즉시 교전
- 아직 표현되지 않은 복합 입장 이벤트와 일부 Room.update 랜덤 분기

이 분기들은 상태 변경 전에 중단하며 Python에 없는 성공 또는 오류 문구를
출력하지 않는다. Rust가 player/mob/item 통합 `room.objs` 삽입 순서를 아직
보존하지 않으므로 서로 다른 객체 종류가 같은 이름으로 동시에 일치할 때는
몹을 임의 우선하지 않는다.
