# 소지품 저장·순서 호환성 근거

검증 기준은 현재 저장소의 Python 런타임 코드뿐이다. Rust에 있던
`inv_stack`/`소지품_수량` 구조를 Python 기능으로 간주하지 않는다.

## Python 데이터 모델

- `objs/object.py:Object.__init__`은 모든 하위 객체를 하나의 `self.objs`
  리스트에 보관한다.
- `Object.insert(obj)`는 같은 객체가 리스트에 없을 때 `obj.env = self`로
  설정하고 인덱스 0에 삽입한다. `append(obj)`만 리스트 끝에 붙인다.
- `objs/player.py:Player.save`는 `for item in self.objs` 순서대로 각 아이템
  객체를 직렬화하고 루트의 `아이템` 배열에 기록한다.
- Python 저장 코드에는 `소지품_수량` 또는 종류별 stack 판정이 없다.
  같은 인덱스와 이름의 아이템도 수량 하나가 아니라 서로 다른 객체다.
- `Player.load`는 `아이템`이 단일 dict이면 한 요소 배열로 바꾼 뒤 배열을
  앞에서부터 순회하고, 복원한 객체마다 `self.insert(obj)`를 호출한다.
  따라서 JSON이 `[A, B, C]`이면 로드 직후 Python `self.objs`는
  `[C, B, A]`이다. 이 역전도 관찰 가능한 원본 동작이므로 유지한다.
- `반응이름`은 로드 시 문자열이면 한 요소 배열로 변환된다. 저장 시에는
  항상 해당 배열을 기록한다. `옵션`과 `아이템속성`은 원본의 문자열/배열
  형태를 유지한다.
- `단일아이템`도 일반 개별 Item 객체다. 저장 레코드에 현재 시각인 `시간`
  필드가 추가되지만, `Player.load`는 이 필드를 읽지 않는다. 소유/버림/보관
  상태는 `objs/oneitem.py:ONEITEM`의 별도 레지스트리 동작이다.
- `호위` 종류만 저장 레코드에 개별 `체력`을 기록한다.
- `Player.load`는 `상태` key가 있는 아이템을 장착 상태로 만들면서 방어력,
  공격력, 장착 무기 참조를 즉시 복원한다. 같은 저장 레코드에 `옵션`도 있을
  때만 힘·민첩성·맷집·체력·내공·필살·운·회피·명중·경험치·마법발견
  보너스를 누적한다. `상태`의 값으로 template의 `계층`을 덮지는 않는다.
- 로그인 `getStart`는 아이템 로드 뒤 `Body.loadSkills()`를 호출해 저장된
  방어무공 효과를 더한다. Rust public loader도 기존 파생 상태를 한 번
  초기화하고 장비를 복원한 뒤 방어무공을 복원해, 같은 Body를 재사용해도
  보너스가 중복되지 않게 한다.

## 순서가 직접 영향을 주는 동작

- `cmds/소지품.py`는 `target.objs`를 순회하면서 이름별 첫 등장 순서를
  유지해 묶어서 출력한다.
- `Object.findObjName`, `findObjInven`, `findObjInUse`와 `Player.getItemIndex`,
  `getItemName`은 리스트 순서에서 N번째 일치 객체를 고른다.
- `cmds/버려.py`, `줘.py`, `가져.py`, `넣어.py`, `꺼내.py`는 얕게 복사한
  현재 리스트 순서와 객체 identity를 이용해 선택·이동한다.
- 구입, 줍기, 받기, 보관함에서 꺼내기 등에서 `insert()`를 호출하므로 새
  아이템은 기존 소지품보다 앞에 놓인다.

## 과거 Rust 형식의 한계와 이관 규칙

과거 Rust는 일부 아이템을 `Object.objs`와 분리한
`HashMap<인덱스, 수량>`에 압축하고 JSON의 `소지품_수량`에 저장했다.
이때 다음 정보는 이미 사라진다.

- 같은 인덱스인 개별 객체마다 다른 이름, 옵션, 속성, 고유번호, 장착 상태
- 서로 다른 stack key 및 `objs` 객체 사이의 실제 취득/삽입 순서
- 같은 key가 다른 아이템 취득 사이에 다시 들어온 interleaving

사라진 순서를 추측해서 복원하지 않는다. 레거시 파일은 다음의 명시적인
일회성 규칙으로만 개별 객체로 확장한다.

1. Python `아이템` 배열은 원본처럼 각 항목을 앞에 삽입한다.
2. 레거시 `소지품_수량` key는 기존 Rust 소지품 표시가 사용하던 key 정렬
   순서로 개별 객체를 만들어 그 뒤에 붙인다.
3. 런타임에 남은 한 stack key를 저장할 때는 Python의 새 아이템 `insert`
   의미에 맞춰 개별 객체들을 기존 리스트 앞에 둔다.
4. 둘 이상의 압축 key가 있으면 key 정렬은 결정적인 이관 순서일 뿐, 잃은
   원래 순서를 복구했다는 의미가 아니다.
5. 데이터 파일이 없어 확장할 수 없는 key는 조용히 버리지 않는다. 레거시
   맵에 남기고 저장을 실패시켜 데이터 손실을 막는다.

새 저장 파일은 Python과 같이 개별 레코드의 `아이템` 배열만 기록해야 하며
`소지품_수량`을 다시 만들지 않는다.

## 회귀 검증

테스트는 실제 `data/user`를 수정하지 않고 메모리와 임시 파일만 사용한다.

- Python 권위 소스가 `self.objs`/`아이템`만 쓰는지 정적 확인
- JSON 배열 로드 역순과 singleton dict 호환
- `반응이름`/`옵션` 배열 및 단일아이템 `시간` 레코드 형태
- 장착 공격력·방어력·무기 참조·옵션 보너스와 방어무공의 1회 복원
- 레거시 수량이 개별 객체 수로 손실 없이 확장되는지 확인
- 복구 불가능한 mixed order를 임의로 호환 완료라고 주장하지 않는 규칙
- Python → Rust load → Rust save → Python load에서 원본과 같은 순서 역전
  및 개별 아이템 필드가 유지되는지 임시 JSON으로 확인
