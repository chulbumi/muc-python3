# MUD Python → Rust/Rhai 마이그레이션 현황

## 현재 판정 (2026-07-10)

마이그레이션은 **진행 중**이다. Python 소스가 런타임 동작의
유일한 기준이며, 해당 동작과 테스트가 대응될 때까지 Python 파일을
삭제하면 안 된다.

현재 `cmds/` 수량은 다음과 같다.

| 형식 | 수량 | 의미 |
|---|---:|---|
| Python `.py` | 190 | 동작 기준 |
| Rhai `.rhai` | 207 | 플레이어 명령 206개와 내부 이동 처리 1개. 파일 존재만으로 이관 완료를 의미하지 않음 |

## 재검증한 범위

다음은 Python 원문과 상태 전이를 직접 대조하고 개별 회귀 테스트를
통과한 범위다.

- Python `objs/alias.py` 런타임 별칭과 정확한 명령 등록 집합
- `말`, `장비`, `무공`, `자동무공`, `쉬어`, `일어나`, `줄임말`
- `전음`/`반전음`, `쪽지`, `귀환`, `끝`/`종료`, `리부팅`, `업데이트`
- `쳐`/`도망` 명령의 검사 순서, 기본 전투 상태, 도망 난수/출구/제한
- 성인 채널 입장/퇴장/잡담/누구와 자동 입장·접속 종료 정리
- 한 단어 방향·고유 출구 이동, 출발/도착 방 알림, 방 보기와 같은 방
  follower의 FIFO 후속 이동(현재 표현 가능한 안전한 방/이벤트 분기)
- `버려`의 선택·전체·수량·50개 방 제한·파괴·ONEITEM 및 Python의
  반복 변수 재사용 동작(개별 객체와 기존 압축 스택 각각)
- 캐릭터 JSON의 일부 Python 호환 로드/저장과 무공 배열 형식
- `조회`/`입금`: 표국무사 검사, 보험료·혜택 계산, 입금 상태 전이를
  Rust efun으로 분리하고 Python 문구·ANSI·CRLF는 Rhai가 담당
- `무공전수`/`무공전수2`/`무공제거`: 같은 방 대상 스냅샷과 네트워크 경계의
  실제 대상 Body 변경 요청을 연결
- `자동경로`: alias 기반 경로 저장·삭제와 Player 경로의 다음 명령 재진입을 연결
- Master apply: `cmds/master.rhai`의 connect/error/create/reset/init 등의
  함수 호출 및 반환값 검증을 연결
- 관리자 아이템 조회: `아이템찾기`, `방어구찾기`, `올숙리스트`를 실제
  `data/item` 카탈로그 기반으로 연결하고 Python 문구·CRLF를 Rhai에서 출력
- 관리자 속성/방 관리: `속성`, `속성추가`, `속성제거`, `방이름`, `방파방이름`을
  대상 속성 조회·추가·부분 제거 및 방 속성 변경으로 연결
- `세트기억`: 장착 중인 무기·방어구의 기존 SET 별칭 제거, 새 SET 별칭 추가,
  `세트기억` 속성 저장과 Python 문구를 연결
- `찾아라`: Python `Room.Zones` 순서에 맞춘 로드 방 순회와 객체명/반응이름
  검색을 `find_object_rooms` efun 및 Rhai 출력으로 연결
- `이벤트`: 현재 방 대상의 `이벤트설정리스트`를 조회해 Python처럼 줄 단위로 출력
- `명칭설정`: 방주 권한·허용 직위명 검증, 길드 명칭 저장 및 길드 공지 연결
- `방파별호`: 같은 방 플레이어·같은 방파 검증, 대상 저장 파일의 별호 갱신,
  ANSI 선포 문구와 방파 공지 연결
- `방파파문`: 길드 역할 목록(문자열/JSON 배열 호환)에서 대상 제거, 방파원수
  감소, 대상 소속·직위·별호 초기화 및 파문 통지 연결
  온라인 대상은 네트워크 경계에서 현재 Body도 즉시 동기화하며 접속은 유지
- `모두끝`: Python의 전역 접속자 목록을 순회해 자신을 제외한 각 사용자에게
  `끝` 명령을 큐잉하도록 연결
- `방설명`: `.` 종료형 `PendingInput`을 추가해 여러 줄 설명을 받고, 현재 방
  런타임 속성과 `data/map/{zone}/{room}.json`의 `맵정보.설명`을 함께 갱신
- `방파방설명`: 방주·부방주 및 방파 소유권 검증 후 동일한 `.` 종료형 방 설명
  입력 상태를 사용
- `방제작`: Python `write_edit`와 동일한 파일 편집 `PendingInput`을 추가해
  `data/map/{zone}/{room}.json`을 줄 단위로 작성
- `아이템제작`, `몹제작`: 동일한 `FileEdit` 상태로 각각 `data/item` 및
  `data/mob/{zone}` JSON 파일을 Python처럼 작성
- `몹회복`: 현재 방 몹의 이름·반응이름을 찾아 최대 체력으로 회복하고
  Python의 성공·실패 문구를 출력
- `몹삭제`: `zone:mob` 인덱스로 런타임 Mob.Mobs 템플릿과 활성 인스턴스를
  제거하며 원본 JSON 파일은 보존
- `아이템삭제`: 런타임 ItemCache 템플릿만 제거하고 원본 `data/item` JSON은
  보존하여 Python `Item.Items` 삭제 의미를 유지
- `방제거`: 로드된 RoomCache에서 `zone:room`만 제거하고 원본 맵 JSON은
  보존하여 Python `Room.Zones` 삭제 의미를 유지
- `사용자몹제거`: 현재 방의 몹 이름·반응이름을 찾아 활성 인스턴스만 제거하고
  몹 템플릿은 보존
- `모두저장`: 네트워크 경계에서 접속 중인 모든 Player Body를 순회해
  Python `Client.players` 일괄 저장을 수행
- `성올려`: 대상 무공의 level과 exp(199999)를 설정하고, 온라인 대상은
  네트워크 경계에서 Body·저장 파일까지 동기화
- `소켓`: 접속자의 peer host·이름을 전역 접속자 스냅샷에서 정렬해 Python
  형식으로 출력
- `정리`: 관리자 1000등급 전용 정리 요청을 전역 접속자에서 확인하고,
  2000등급 `Kick`과 분리된 연결 종료 경로로 처리
- `세트착용`: `세트기억` 별칭이 붙은 미착용 무기·방어구를 조회해 기존
  `item_equip` 경로로 순서대로 장착
- `옵설정`: 소지품 대상의 옵션을 갱신하고 Python의 파란색 이름 표시와
  성공·실패 문구를 연결
- Python `MagicMap` 18개 계층별 옵션 상한을 `data/config/magic_map.json`으로
  추출하고 `옵랜덤`의 확률·옵션·공격/방어력·색상·보호 속성 적용을 연결
- `방주권한양도`: 부방주 대상·소속·레벨 검증, 길드 역할 목록 이동, 방주/부방주
  저장 속성 갱신 및 Python 형식의 길드 공지 연결. 온라인 대상 Body도 즉시 동기화
- `지워지워`: 소지품 아이템의 옵션을 출력하고 `아이템속성`·`옵션`을 삭제하는
  Python 동작을 연결
- `입문신청`: 같은 방 방주·부방주 대상의 소속·직위·성격·중복 신청을 확인하고
  대상 저장 데이터의 `입문신청자`와 신청 통지를 처리
- `누가주나`: 로드된 Mob 템플릿의 판매·사용 아이템 목록을 순회해 지정 아이템을
  가진 몹을 출력

같은 방 플레이어 조회는 전체 접속자를 순회하지 않는다.
`WorldState` 방별 삽입 순서 인덱스에서 이름을 얻고,
`Broadcaster` 이름 → 접속 인덱스로 해당 세션만 직접 조회한다.
`get_all_online_players()`는 Python도 전체 접속자를 보는 `누구`, `어디`에만
남겨 두었다.

## 아직 완료되지 않은 부분

- 설치 보관함·바닥 아이템이 있는 방처럼 통합 `room.objs` 순서가 필요한
  이동, 치명적 함정, 복합 입장 이벤트·mob update/선공 분기
- 매 틱 전투, 자동/몽 무공, 사망·혼수·전투 후 처리
- `따라`, `무리`, `무리제외`, `무리말` 런타임 관계
- `넣어`, `꺼내` 보관함 전체 동작과 일부 아이템 명령
- Python `아이템` 배열과 Rust 압축 스택 사이의 통합 소지품 삽입 순서 및
  양방향 저장 형식
- 방파/관리자 명령의 대상 상태 전이와 일부 대화형 입력
- 객체별 `lib/std/*.rhai` lifecycle 및 heartbeat의 실제 GameLoop 등록/호출 wiring
- Python의 player/mob/item/box 통합 `room.objs` 삽입 순서
- `Room.update`, 복합 입장 이벤트, 자동 이동, 치명적 함정의 전체 결과

표시만 되는 `현재 구현 중`, `미구현`, `TODO` 응답은 성공으로
계산하지 않는다. Python에 없는 안내문을 만드는 대신 해당 기능을
이관하고 정확한 출력·상태 변화를 테스트해야 한다.

## 완료 판정 조건

1. 모든 Python 명령과 파서 특수 경로를 같은 상태/입력에서 비교한다.
2. 문구·ANSI·CRLF·prompt 순서와 수치/상태 변화가 같아야 한다.
3. Python ↔ Rust 캐릭터/아이템/방 데이터를 양방향으로 로드한다.
4. placeholder, Python에 없는 명령/별칭/출력, 구조적 미이관 분기가 0건이어야 한다.
5. 단위·통합·TCP 비교 테스트와 Rust 정적 검사가 모두 통과해야 한다.

그 전에는 이 저장소의 Python 서버와 소스를 비교 기준으로 보존한다.
* `방파입문`: same-room 신청자 검증 후 Python의 제자 목록/방파원수 갱신과 대상 속성 저장을 수행하도록 `guild_add_member` efun으로 이관했다.
* `호위`: Python의 호위 목록/체력 비율/분노/설명 출력을 `get_guards` efun 기반 Rhai 명령으로 연결했다.
* `현판걸어`: Python의 방파자리/방파주인·방파금지·은전·중복명 검사를 수행하고, 길드 생성·현판 속성·보관함 지급·은전 차감·공지까지 Rhai/efun으로 연결했다.
* `조제`: 의원 몹의 `조제 이름` 이벤트를 `get_room_mobs`로 조회하고, 재료 보유량을 선검증한 뒤 소모·결과물 생성하도록 Rhai/efun으로 이관했다.
* `지연입력`: `call_out` 기반 1초 지연 후 동일 플레이어 명령 큐에 넣도록 구현했다.
* `모두소환`: Python의 전역 접속자 순회를 유지하되 현재 방은 제외하고 `warp_player`로 관리자 소환을 수행하도록 연결했다.
* `remove_user_mob` efun: 관리자 사용자 몹 검색을 전체 활성 인스턴스에서 수행하고 실제 인스턴스를 제거하도록 연결했다.
* `줘줘`: 관리자 다중 지급 명령을 기존 `request_give_silver`/`request_give_item` 전달 경로에 연결했다.
* `분해`: 상인(`물건구입`) 존재·아이템 속성·장비 상태를 Rust efun에서 검사하고 Python의 옵션 수 기반 강철조각 생성량과 인벤토리 상태 전이를 구현했다.
* `get_room_players_admin`: 이름만 반환하던 임시값을 제거하고 Python 사용자 JSON의 레벨·체력·내공·힘·민첩성 속성을 관리자 조회에 반영했다.
* 관리자 `set_player_level`·`set_player_money`·`set_player_hp`가 로그만 남기지 않고 대상 사용자 JSON의 실제 속성을 갱신하도록 수정했다. 온라인 Body 동기화는 별도 broadcaster 요청 경로가 필요하다.
* 관리자 플레이어 속성 설정은 파일 저장뿐 아니라 접속 중 대상 Body에도 `SET_PLAYER_ATTR_REQUEST`로 동기화하도록 연결했다.
* `give_item_to_mob`: `MobInstance.inventory`를 추가해 스택·개별 아이템 모두 실제 몹 인벤토리로 이동하도록 구현했다(기존 삭제-only TODO 제거).
* `get_room_mobs`가 몹의 런타임 인벤토리 이름 목록도 노출해 Rhai 이벤트/관리자 조회에서 전달된 아이템 상태를 확인할 수 있게 했다.
* `comm.rhai`: Python `comm.py`의 follower/group 제외 동작을 Party membership efun(`party_request_remove_followers`)으로 연결했다.
* Party 스냅샷에 `_combat_target_ids`를 `combat_target_ids` 배열로 노출해 향후 방어지정·합동공격의 네트워크 대상 상태를 전달할 기반을 추가했다.
* `SocialState`에 connection별 `combat_targets`와 `SetCombatTargets` 요청을 추가하고 `party_request_set_combat_targets` efun으로 노출해 방어지정·합동공격의 네트워크 상태 기반을 마련했다.
* `방어지정`: Party leader가 전투 중 구성원의 `combat_target_ids`를 선택해 방어 대상 상태 요청을 각 구성원에게 전달하도록 연결했다.
* `업데이트` hot-reload 경로를 재검증하고, 새 `방어지정` 스크립트의 strict Rhai 컴파일 오류를 수정해 명령어 reload 회귀를 복구했다.
* `무리합`: 방장이 자신의 무리 전투 대상 또는 전투 중인 무리원의 `combat_target_ids`를 전체 무리원에게 전파하도록 구현했다.
* Party 전투 대상 전파에 대해 방장·무리원 권한 범위를 검증하는 회귀 테스트를 추가했다.
* 최신 검증: `cargo test --all-targets` 단위 496개와 통합 11개 전체 통과, Python `9903`·Rust `9999` 동시 기동 확인.
